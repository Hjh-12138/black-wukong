# 黑神话悟空领域大模型微调项目文档

## 一、项目意义

本项目旨在通过**知识蒸馏 + QLoRA 微调**两阶段流水线，将通用大语言模型（Qwen3-4B-Instruct）适配为《黑神话：悟空》游戏领域的专业问答助手。

### 核心价值

1. **低成本领域适配**：在消费级 GPU（RTX 4070 SUPER 12GB）上完成 4B 参数模型的微调，无需企业级硬件即可实现高质量的领域知识注入。
2. **自动化知识提取**：通过教师模型（DeepSeek V4 Flash）自动从非结构化游戏攻略文档中提取结构化 Q&A 对，大幅降低人工标注成本。
3. **可复现的流水线**：整个流程从原始 MD 文档到可部署的推理模型，形成标准化、可复用的技术方案，可迁移至其他垂直领域。

### 应用场景

- 游戏智能助手：实时回答玩家关于武器、防具、道具、BOSS 攻略等问题
- 垂直领域知识库构建：为其他游戏或专业领域提供参考方案
- 边缘设备部署：量化后的 LoRA 适配器仅 132MB，可在消费级设备上运行

## 三、结论

### 训练结果

| 指标 | 数值 |
|------|------|
| 训练样本数 | 1256 |
| 验证样本数 | 72 |
| 训练轮次 | 4 epochs |
| 学习率 | 3e-4 (linear decay) |
| 最终训练损失 (train loss) | 0.7535 |
| 最佳验证损失 (eval loss) | 1.674 (@ checkpoint-150) |
| 训练时长 | ~43 分钟 |
| LoRA 适配器大小 | 132 MB |

### 推理效果

模型能够正确回答《黑神话：悟空》领域内的问题，包括：

- **武器获取**："如何获得出云棍？" → 准确描述获取方式
- **BOSS 位置**："黑风大王在哪里打？" → 准确给出位置信息
- **技能升级**："定身术怎么升级？" → 给出升级条件和方法
- **道具收集**："盘丝岭有哪些重要宝物？" → 列出宝物清单

### 关键发现

1. **4B 模型 + QLoRA 足以承载中等规模的领域知识**：1256 条高质量样本即可使模型获得显著的领域问答能力。
2. **过拟合问题值得关注**：训练损失持续下降至 ~0.01，但验证损失在 checkpoint-150 后开始上升，说明模型在后期出现了明显的过拟合。最佳模型出现在第 150 步而非最终步。
3. **数据质量 > 数据数量**：通过教师模型生成的多角度 Q&A（每段 3 个不同角度的问题）比单纯的同义改写能提供更丰富的训练信号。
4. **消费级 GPU 完全可行**：12GB 显存 + 4-bit 量化 + LoRA 的组合在训练速度和显存占用之间取得了良好平衡。

### 技术创新点

1. **Windows 平台 safetensors mmap 兼容方案**：通过自定义 `_NoMmapSafeOpen` 类预先将 safetensors 数据读入内存，绕过 Windows 对 mmap 的限制，使训练脚本可在 Windows 上稳定运行。
2. **智能中文文档切分**：针对中文游戏攻略的格式特点（h2/h3/h4 标题 + 粗体关键词），设计了分层次的文档切分策略，确保每个切分段都具有独立的信息完整性。
3. **多角度 Q&A 生成**：教师模型对每段文本从"是什么""怎么获得""在哪里""有什么效果"等不同角度生成 3 个问题，大幅提升训练数据的覆盖度。

## 五、项目中遇到的问题

### 1. Windows 平台 safetensors 文件加载失败

**问题描述**：Windows 不支持 Linux 的 `mmap` 系统调用，transformers 默认的 `safetensors.safe_open` 在 Windows 上加载模型权重时会失败。

**解决方案**：在训练脚本中实现了自定义的 `_NoMmapSafeOpen` 和 `_TensorSlice` 类，将 safetensors 文件的全部字节数据预先读入内存，再通过 `torch.frombuffer` 重建张量，完全绕过 mmap 机制。同时通过 monkey-patch 替换 `modeling_utils.safe_open` 和 `modeling_utils.load_state_dict` 函数。

**关键代码**：`千问 3.0 qlora_v4.py` 第 16-86 行。

### 2. 模型缓存路径未配置导致重新下载

**问题描述**：训练脚本默认设置 `HF_HUB_OFFLINE=1` 且未配置 `HF_HUB_CACHE`，导致无法找到已缓存在 `D:\models\hub\hub\` 下的模型文件。

**解决方案**：在运行训练和推理脚本时，显式设置 `HF_HUB_CACHE=D:/models/hub/hub` 环境变量，使 HuggingFace Hub 能正确索引到本地缓存的 Qwen3-4B 模型。

### 3. OpenMP 动态库重复加载冲突

**问题描述**：在同一进程中，PyTorch 和 NumPy 各自加载了不同版本的 `libiomp5md.dll`，导致 `OMP: Error #15` 运行时错误。

**解决方案**：设置环境变量 `KMP_DUPLICATE_LIB_OK=TRUE` 允许重复加载，作为临时兼容方案。生产环境中建议统一依赖版本彻底解决。

### 4. 模型过拟合

**问题描述**：训练过程中训练损失快速下降（最终 ~0.75），但验证损失在第 150 步（约 1.5 个 epoch）达到最低点 1.674 后开始反弹上升，到第 600 步时已升至 ~2.29，出现明显的过拟合。

**原因分析**：
- 训练数据量相对较小（1256 条），模型容量（4B）相对充足
- Paraphrase 生成的变体问题与原问题语义高度相似，实际增加了数据的冗余度而非多样性
- 4 个 epoch 对于当前数据量偏多

**缓解措施**：
- 设置 `load_best_model_at_end=True`，自动加载验证损失最低的检查点
- 使用 checkpoint-150 作为最终推理模型
- 后续改进方向：增加数据多样性、降低训练轮次、增大 LoRA dropout

### 5. 教师模型 API 调用不稳定

**问题描述**：DeepSeek API 在批量调用过程中偶发失败（部分 segment 返回 0 条 Q&A，或返回 JSON 格式不完整），导致数据生成不完全。

**解决方案**：
- 实现了 3 次指数退避重试机制 (`time.sleep(1.5 ** attempt + random.random() * 0.3)`)
- 对返回结果进行严格的格式校验（问题长度 ≥ 8，答案长度 ≥ 10）
- 容忍部分数据丢失（270 段中少数段未生成 Q&A），不影响整体数据量

### 6. 推理脚本后台运行时陷入 EOF 死循环

**问题描述**：推理脚本包含交互式循环 (`while True: input(...) `)，在后台运行时 stdin 为空，导致无限 EOF 循环。

**解决方案**：重构推理脚本为"批量测试 + 可选交互"模式，默认执行批量测试后自动退出，仅在 stdout 为 TTY 时才进入交互模式。同时支持命令行参数直接传入问题进行单次推理。

### 7. API Key 硬编码安全风险

**问题描述**：`多md的教师模型_v4.py` 第 29 行直接硬编码了 DeepSeek API Key (`sk-24a4382641974bbcbbacc58c819e143c`)。

**风险说明**：如果将代码提交到公开仓库，API Key 会直接暴露，可能导致密钥被滥用或泄漏。

**建议改进**：使用环境变量或 `.env` 文件管理敏感凭证。

### 8. 推理脚本 checkpoint 路径过期

**问题描述**：推理脚本中的 `lora_adapter_path` 指向旧的训练运行目录 (`20260505_230052`)，而实际使用的是新训练的 checkpoint (`20260506_155303`)，导致推理脚本无法加载模型。

**解决方案**：更新推理脚本中的路径配置，使其指向最新训练运行的 checkpoint-150（最佳模型）。
